## 任务
给出你的诊断。务必包含以下部分：

### 1. 现象（Facts）
你具体看到了什么异常？引用关键日志或指标。

### 2. 假设（Hypothesis）
你认为根因是什么？可以给出多个假设并排序。

### 3. 把握（Confidence）
你对最可能的假设有多大把握？用百分比表示。

### 4. 缺失信息（Gaps）
如果把握不够（<70%），你还需要什么信息？**给出具体的 shell 命令**放在 ```commands 代码块中。
Agent 会自动执行这些命令并用结果重新诊断——所以尽管列出你需要的命令。

### 5. 是否需要人类（Escalate）
YES 或 NO。
- **只在以下情况回答 YES**：涉及客户数据安全、需要商业决策、需要物理操作（如换硬件）、或需要外部供应商介入
- **以下情况回答 NO**：信息不足（在 Gaps 中列出需要的命令即可）、代码 bug（可以通过补丁修复）、配置错误（可以修改配置）、资源问题（可以扩容/清理/重启）、外部依赖问题（可以重试/切换/降级）
- 记住：你有能力执行命令收集更多信息、生成代码补丁、重启服务、修改配置。遇到技术问题先自己想办法。

### 6. 类型（Type）
单选一个最贴切的值,只输出关键词(不要解释):
- `code_bug` — 应用代码层面的 bug,可以通过修改源码修复(空指针、越界、逻辑错误等)
- `runtime` — 进程崩溃/卡死/死锁,通常重启可以暂时缓解
- `config` — 配置文件错误、环境变量缺失、权限问题
- `resource` — 磁盘满、OOM、CPU 打满、文件描述符耗尽
- `external` — 外部依赖故障(数据库连不上、上游 API 503、DNS 失败)
- `unknown` — 信息不足以判断

**重要**:只有 `type: code_bug` 且源码上下文里有具体代码片段时,Agent 才会进入自动补丁流程。

## 重要
- 不确定就说不确定，不要硬猜——但**不确定不等于需要人类**，把缺失的命令列在 Gaps 中
- 宁可多收集信息，不要贸然行动
- 你的目标是**自主解决问题**，只在真正超出技术范畴时才 Escalate
