## 1. Installation & Dependencies

**System requirements:**
- Python ≥ 3.9
- git ≥ 2.20 (local repos + patch application)
- Optional: `gh` CLI (GitHub PR workflow) / `glab` CLI (GitLab)
- Optional: `docker` / `kubectl` (for corresponding target types)

**Python dependencies** (`requirements.txt`):

```
anthropic>=0.40.0
openai>=1.50.0
prompt_toolkit>=3.0.0
pyyaml>=6.0
```

Only these 4 third-party packages — everything else in Sprint 2-6 uses stdlib (stack parsing, git operations, HTTP health checks, IM notifications, and Prometheus metrics are all hand-written with zero dependencies).

```bash
git clone <repo-url> && cd ops-agent
pip install -r requirements.txt
```

**Install the `ops-agent` command (recommended):**

```bash
# Option 1: pip editable install (gives you the ops-agent command)
pip install -e .

# Option 2: Use the wrapper script directly
./scripts/ops-agent --help
```

After install, all `ops-agent` commands in the examples work. Without install, substitute `python main.py`.

### 2.1 LLM Configuration

Configure via environment variables. Three providers supported:

```bash
# Anthropic Claude (default, recommended)
export OPS_LLM_PROVIDER=anthropic
export OPS_LLM_API_KEY="sk-ant-..."
export OPS_LLM_MODEL=claude-sonnet-4-20250514     # Optional, has default

# OpenAI
export OPS_LLM_PROVIDER=openai
export OPS_LLM_API_KEY="sk-..."
export OPS_LLM_MODEL=gpt-4o

# Zhipu GLM
export OPS_LLM_PROVIDER=zhipu
export OPS_LLM_API_KEY="..."
export OPS_LLM_MODEL=glm-4-plus

# Any OpenAI-API compatible local model (Ollama / vLLM / LM Studio)
export OPS_LLM_PROVIDER=openai
export OPS_LLM_BASE_URL=http://localhost:11434/v1
export OPS_LLM_MODEL=llama3
export OPS_LLM_API_KEY=dummy
```

**LLM failure handling**:
All LLM calls are wrapped in `RetryingLLM` by default, which automatically retries 3 times on API flakiness (exponential backoff 1s/2s/4s). Consecutive failures → raises `LLMDegraded` → Agent auto-switches to readonly + escalates to human + retries every 5 minutes. Auto-recovers when LLM comes back.

### 2.2 Target Configuration targets.yaml

`notebook/config/targets.yaml` defines all target systems the Agent manages.

```yaml
targets:
  # ── Local ──
  - name: local-dev
    type: local
    description: Local dev machine patrol
    criticality: low
    tags: [dev]

  # ── SSH Remote ──
  - name: web-prod-01
    type: ssh
    host: ops@web01.example.com
    port: 22
    key_file: ~/.ssh/id_ed25519
    description: Production web node
    criticality: critical
    tags: [prod, web, frontend]

  # ── Docker Host ──
  - name: dockerhost-staging
    type: docker
    docker_host: ""               # Empty = local unix socket
    compose_file: /opt/staging/docker-compose.yml
    description: Staging docker compose
    criticality: normal
    tags: [staging]

  # ── Kubernetes Cluster ──
  - name: k8s-prod
    type: k8s
    kubeconfig: ~/.kube/config
    context: prod-cluster
    namespace: backend
    criticality: critical
    tags: [prod, k8s]
```

Each target has its own `ToolBox` (command executor) but shares the same `Notebook` and `LLM`.

The **`criticality`** field tells the LLM the target's importance in the prompt, influencing how cautiously the Agent behaves.

The **`tags`** field is used for playbook filtering during retrieval (e.g., a playbook that only applies to targets tagged `[prod]`).

### 2.3 Source Repositories SourceRepo

To enable source code bug auto-fix (Sprint 2-4 capabilities), add `source_repos` under each target:

```yaml
targets:
  - name: web-prod-01
    type: ssh
    host: ops@web01.example.com
    source_repos:
      - name: backend
        path: /opt/sources/backend            # Local clone on the Agent workstation
        repo_url: git@github.com:org/backend.git
        branch: main
        language: python                       # python / java / go / node / rust / cpp
        build_cmd: "python -m py_compile $(git ls-files '*.py')"
        test_cmd: "pytest -x --timeout=30"

        # Path prefix mapping from runtime entity to local clone
        # In container: /app/handlers/user.py, locally: /opt/sources/backend/handlers/user.py
        path_prefix_runtime: /app
        path_prefix_local: ""

        # ── Sprint 4: PR workflow ──
        git_host: github                       # github | gitlab | noop
        base_branch: main
        deploy_signal:
          type: http
          url: http://web01.example.com/version
          expect_contains: "{commit_sha}"
          check_interval: 10
          timeout: 1800

        # Log path for recurrence detection (optional)
        log_path: /var/log/backend/error.log
```

**Field descriptions:**

| Field | Required | Description |
|---|---|---|
| `name` | ✓ | Repo alias, used in logs and PR titles |
| `path` | ✓ | Absolute path to local clone (Agent must have read/write access) |
| `language` | Recommended | Soft filter for stack_parser and source_locator |
| `build_cmd` | Required for patch fix | Build/syntax check command, 5-minute timeout |
| `test_cmd` | | Unit test command, 10-minute timeout. Empty = skip tests |
| `path_prefix_runtime` | | Path prefix inside the container, e.g. `/app` |
| `path_prefix_local` | | Relative prefix within the local clone (usually empty) |
| `git_host` | Required for auto-PR | `github` / `gitlab` / `noop` (local verify only, no push) |
| `base_branch` | | PR target branch, default `main` |
| `deploy_signal` | | See [§5.2](#52-deploy-signal-configuration) |

**Incremental enablement recommendation**:
- Phase 1: Configure only `path` + `language` to enable Sprint 2 source code location (Agent writes "error at user.py:42" in incident notes)
- Phase 2: Add `build_cmd` + `test_cmd` to enable Sprint 3 local patch verification (Agent generates patches but doesn't push)
- Phase 3: Add `git_host: github` + `deploy_signal` to enable Sprint 4's full auto-PR/merge/observation/revert flow

### 2.4 Authorization Rules permissions.md

`notebook/config/permissions.md` uses natural language to tell the Agent which L2+ actions can be executed automatically and which require human confirmation. Example:

```markdown
# Authorization Rules

## Auto-approved (L2 automatic)
- Restart nginx / php-fpm / redis (max 3 times per day)
- Clean up files older than 7 days in /tmp and /var/log
- Restart docker containers (subject to limits.yaml max_restarts_per_service_per_day)

## Must ask human (L2 but requires confirmation)
- Modify any config files under /etc (must cp backup before changing)
- Scale K8s deployments
- Any operations involving databases (mysql/postgres)

## Always forbidden (L4)
- rm -rf /
- DROP TABLE / TRUNCATE
- Shutdown/reboot the server
- Modify /etc/passwd / /etc/shadow
```

The Agent reads this file during the plan phase, and the LLM judges which tier the current action falls into. The blacklist (L4) is hard-coded as a backstop in `src/safety/safety.py` — it blocks regardless of what the LLM says.

### 2.5 Blast Radius limits.yaml

`notebook/config/limits.yaml` is a physical guardrail the LLM cannot bypass:

```yaml
enabled: true

# Action frequency
max_actions_per_hour: 20
max_actions_per_day: 100

# Per-service limits
max_restarts_per_service_per_day: 5
max_restarts_per_service_per_hour: 3

# Concurrency cap
max_concurrent_incidents: 2

# Failure cooldown (how long before retry is allowed after a fix failure)
cooldown_after_failure_seconds: 600

# LLM cost
llm_tokens_per_day: 1000000
llm_tokens_per_hour: 200000

# Sprint 4: Auto-merge PR cap
max_auto_merges_per_day: 5
```

Any breach forces escalation to a human — the Agent cannot override these for any reason.

### 2.6 IM Notifications notifier.yaml

`notebook/config/notifier.yaml` (see `notifier.yaml.example`):

```yaml
type: slack                # slack | dingtalk | feishu | feishu_app | none
webhook_url: https://hooks.slack.com/services/XXX/YYY/ZZZ

notify_on:
  - incident_opened
  - incident_closed
  - pr_merged
  - revert_triggered
  - critical_failure
  - llm_degraded
  - daily_report

quiet_hours:
  start: "22:00"
  end: "08:00"
  except_urgency:
    - critical
```

**Security best practice**: Override `webhook_url` via environment variable to avoid credentials in git:

```bash
export OPS_NOTIFIER_WEBHOOK_URL="https://hooks.slack.com/services/..."
```

`quiet_hours` supports both cross-day (`22:00 → 08:00`) and same-day (`13:00 → 14:00`) ranges. `critical` urgency notifications bypass quiet hours.

#### Feishu Configuration

##### Option 1: Webhook Bot (simple, recommended for beginners)

1. In your Feishu group chat, click "Settings → Group Bots → Add Bot → Custom Bot"
2. Copy the Webhook URL (format: `https://open.feishu.cn/open-apis/bot/v2/hook/xxx`)
3. Configure `notifier.yaml`:

```yaml
type: feishu
webhook_url: "https://open.feishu.cn/open-apis/bot/v2/hook/xxx"
```

Or run `ops-agent init`, select `feishu` as notification type, and enter the Webhook URL.

> Webhook bots can only push messages to group chats; they cannot receive replies.

##### Option 2: Custom App Bot (full-featured, supports two-way interaction)

**Prerequisites**: Feishu Open Platform admin access.

**Step 1 — Create an app**

1. Log in to the [Feishu Open Platform](https://open.feishu.cn/app), click "Create Enterprise Custom App"
2. Fill in the app name (e.g., `OpsAgent`) and description
3. Record the **App ID** (`cli_xxx`) and **App Secret**

**Step 2 — Add bot capability**

1. Go to the app → "Add App Capability" → check "Bot"
2. Under "Permission Management", search for and enable:
   - `im:message:send_as_bot` (send messages as bot)
   - `im:message` (get & send chat and group messages)
   - `im:chat` (get group info)

**Step 3 — Get chat_id**

1. Create a Feishu group chat (or use an existing one for Agent notifications)
2. Add the bot to the group (search for the app you just created)
3. Get the `chat_id`: Group settings → Group card → Copy group link; the string after `/chat/` in the URL is the `chat_id` (format: `oc_xxx`)

**Step 4 — Configure notifier.yaml**

```yaml
type: feishu_app
feishu_app:
  app_id: "cli_xxxxxxxx"
  app_secret: "xxxxxxxxxxxxxxxx"
  chat_id: "oc_xxxxxxxx"
```

**Security tip**: Don't commit `app_secret` to git — override it with environment variables:

```bash
export OPS_FEISHU_APP_ID="cli_xxxxxxxx"
export OPS_FEISHU_APP_SECRET="xxxxxxxxxxxxxxxx"
export OPS_FEISHU_CHAT_ID="oc_xxxxxxxx"
```

**Step 5 — Enable two-way interaction (optional)**

Once enabled, you can @mention the bot in Feishu group chats and the Agent will receive and reply:

1. Add `interactive` config to `notifier.yaml`:

```yaml
type: feishu_app
feishu_app:
  app_id: "cli_xxxxxxxx"
  app_secret: "xxxxxxxxxxxxxxxx"
  chat_id: "oc_xxxxxxxx"
  interactive:
    enabled: true
    callback_port: 9877       # Feishu event callback port
    encrypt_key: ""           # Open Platform → Event subscription → Encrypt Key
    verification_token: ""    # Open Platform → Event subscription → Verification Token
```

2. Configure event subscription on the Feishu Open Platform:
   - Request URL: `http://<your-public-ip>:9877/feishu/event`
   - Subscribe to event: `im.message.receive_v1`

3. Ensure port 9877 on your server is reachable by Feishu servers (requires public IP or reverse proxy)

> Two-way interaction currently only supports the Feishu custom app mode (`feishu_app`); Webhook bots do not support it.

### 3.2 Crash Recovery

OpsAgent checkpoints state to `notebook/state.json` (atomic write) after every main loop iteration. This includes:

- Current mode (patrol / investigate / incident)
- Current target / incident
- readonly / paused flags
- Last error text (used as recurrence detection baseline)
- Auto-merge PR timestamps (for rate limit reset)

**Automatic recovery on restart**. If an unfinished incident is detected, the Agent will:

1. Print "⚠️ Detected unfinished work from last session, state recovered"
2. **Not replay unfinished actions** (interrupted actions may have been partially executed; replaying could cause harm)
3. Let the human decide whether to continue or abandon

State file version mismatch / corrupted / missing → fresh start; the Agent never crashes due to corrupted state.

### 3.3 Watchdog Integration

After startup, the Agent exposes a health endpoint on `127.0.0.1:9876` by default. An external watchdog can actively probe:

```bash
# Simple cron probe
* * * * * /opt/ops-agent/scripts/watchdog.sh

# scripts/watchdog.sh built-in logic:
#   3 consecutive health check failures → systemctl restart ops-agent
```

Also works as a K8s liveness probe:

```yaml
livenessProbe:
  httpGet:
    path: /healthz
    port: 9876
  initialDelaySeconds: 30
  periodSeconds: 30
  failureThreshold: 3
```

## 4. Talking to the Agent

Once the Agent is running, you can type messages in the terminal at any time. The Agent can be interrupted during any operation (including LLM streaming generation, running commands) and responds within seconds.

### Built-in Commands

| Command | Description |
|---|---|
| `status` | View Agent's current state (mode, target, incident, limit quotas) |
| `pause` / `resume` | Pause/resume autonomous patrol (Agent still responds to your commands) |
| `stop` | Stop current investigation/incident, return to patrol |
| `readonly on` / `readonly off` | Toggle read-only mode |
| `target <name>` | Switch the active target (if multiple are configured) |
| `quit` / `exit` | Exit the Agent |

### Natural Language Conversation

```
> Any nginx 5xx errors in the last hour?
> Check the disk usage on db-prod for me
> What do you think caused this incident?
> Don't restart redis, let me check first
```

The Agent treats these as "human instructions" and acts immediately (reads logs, runs commands, answers questions) rather than treating them as casual conversation.

### Emergency Stop

Press `Ctrl-C` or type `stop` at any time — the Agent will immediately:
1. Interrupt any in-progress LLM streaming generation
2. Interrupt any running SSH/Docker commands
3. Write "interrupted by human" in the incident notes
4. Return to patrol mode and wait for the next instruction

## 5. Autonomous Repair Loop

### 5.1 Full Code Bug Auto-Fix Flow

Complete flow (Sprint 2-4 capabilities):

```
Anomaly detected
  │
  ├── observe → assess → "This is abnormal"
  │
  ├── diagnose: LLM analysis + auto-locate anomaly source code          [Sprint 2]
  │     ↳ stack_parser parses Python/Java/Go/Node tracebacks
  │     ↳ source_locator reverse-traces to local clone
  │     ↳ Extracts target lines ±30 lines context + function definition
  │     ↳ Injected into diagnose prompt's {source_locations}
  │     ↳ LLM outputs diagnosis conclusion with type=code_bug
  │
  ├── Trigger conditions met (type=code_bug + has source_repos + not readonly + has build_cmd)
  │     ↓
  ├── PatchLoop (max 3 retries)                                          [Sprint 3]
  │     │
  │     ├── PatchGenerator: LLM generates unified diff
  │     ├── PatchApplier:
  │     │     1. git stash (dirty working directory)
  │     │     2. git checkout -b fix/agent/<incident-id>
  │     │     3. git apply
  │     │     4. git commit
  │     │     5. build_cmd (5min timeout)
  │     │     6. test_cmd (10min timeout, optional)
  │     │     7. Any failure → reset --hard + clean -fd + branch -D + stash pop
  │     │
  │     └── Failure → feed diff and error info back to LLM for retry
  │
  ├── All 3 attempts fail → escalate to human, local workspace completely clean
  │
  └── Success → VerifiedPatch
        ↓
        Sprint 4 PR workflow (if git_host is configured)
        │
        ├── Rate limit check (max_auto_merges_per_day)
        ├── git push -u origin <branch>
        ├── gh pr create
        ├── Re-check PR CI status (failure → downgrade, wait for human)
        ├── gh pr merge --squash --delete-branch
        ├── DeployWatcher waits for deploy signal
        ├── ProductionWatcher observes for 5 minutes
        │     ↳ Uses ParsedTrace.signature() for exact recurrence matching
        │
        ├── No recurrence → close incident, IM notification ✅
        └── Recurrence → RevertGenerator auto git revert
              + create revert PR + immediate merge
              + escalate to human ⚠️
```

**Safety constraints** (each layer independently acts as a backstop):

| Layer | Constraint |
|---|---|
| PatchGenerator | Only accepts standard unified diffs with `@@` and `+++` lines; otherwise treated as generation failure |
| PatchApplier | Rejects patches that only modify test files (`Patch.touches_only_tests()` cheat guard) |
| PatchApplier | Any stage failure = `git reset --hard` + `git clean -fd` + `git branch -D` + `git stash pop` — workspace absolutely clean |
| PatchLoop | Max 3 retries; all failures count toward limits quota |
| Sprint 4 | Re-checks CI before merge; CI red → downgrade, wait for human |
| Sprint 4 | `max_auto_merges_per_day=5` default cap |
| Sprint 4 | Recurrence detection uses strict signature matching (language:type:file:line); no fuzzy matching to avoid false positives |
| Sprint 4 | Auto-revert also counts as an auto-merge, preventing vicious loops |

### 5.2 Deploy Signal Configuration

`deploy_signal` tells the Agent "how to know my patch has been deployed to production". Supports 4 types:

```yaml
# 1. HTTP probe: GET a URL, response contains commit_sha
deploy_signal:
  type: http
  url: http://prod.example.com/version
  expect_contains: "{commit_sha}"   # Auto-replaced with actual commit
  check_interval: 10                 # Probe every 10 seconds
  timeout: 1800                      # Max wait 30 minutes

# 2. File signal: a file exists and contains commit_sha
deploy_signal:
  type: file
  path: /var/run/deploy/version.txt
  expect_contains: "{commit_sha}"

# 3. Command signal: run a command, exit 0 = deployed
deploy_signal:
  type: command
  cmd: "kubectl get deploy backend -o jsonpath='{.metadata.annotations.commit}' | grep {commit_sha}"

# 4. Fixed wait: simply wait N seconds (no CD environment)
deploy_signal:
  type: fixed_wait
  seconds: 60
```

No `deploy_signal` configured → assumes deployed by default, skips waiting and goes directly to observation period.

### 5.3 Production Observation Period & Auto-Revert

After deployment, the Agent enters a **5-minute observation period** (`duration` parameter is adjustable in code). Every 30 seconds it calls `observe_fn` (defaults to tailing 200 lines from `repo.log_path`) and checks for a **matching signature** recurrence using `ParsedTrace.signature()`.

**4 possible outcomes:**

| Outcome | Meaning | Follow-up |
|---|---|---|
| `OK` | Observation period ended, no recurrence | ✅ Close incident, IM notification of success |
| `FAILED_RECURRENCE` | Original anomaly recurred | ⚠️ Trigger RevertGenerator, auto-revert + escalate |
| `OBSERVE_ERROR` | observe_fn failed 3 consecutive times | Escalate to human ("I merged but can't see production") |
| `NO_BASELINE` | Cannot extract signature from original incident | Escalate to human ("Can't do recurrence detection, need manual confirmation") |

**Revert failure has a backstop too**: If RevertGenerator itself fails → immediately escalate to human ("revert also failed, human needs to intervene immediately").

### 6.3 IM Notifications & Daily Report

```

   ↓

   ↓
```

Send failures don't block the main loop (only logged via logger.warning).

**Daily report**: `reporter.py` checks `should_send_today()` in the main loop. Deduplication uses a `marker_dir/sent-YYYY-MM-DD` empty file marker, persistent across processes.

## 7. Security & Emergency Stop

### Trust Levels

| Level | Description | Who approves |
|---|---|---|
| **L0** | Read-only observation (tail/grep/cat/ps/df...) | Automatic |
| **L1** | Write notes (markdown within notebook) | Automatic |
| **L2** | Service operations (restart/config change) | permissions.md rules |
| **L3** | Code changes (git apply/PR) | Rate limiting + auto-verification |
| **L4** | Destructive operations (`rm -rf /` / `DROP TABLE` / `mkfs` / ...) | **Always forbidden**, hard-coded blacklist |

### Three Ways to Trigger Emergency Stop

**1. File trigger:**

```bash
# Create file → Agent detects on next loop → immediately read-only
touch /var/run/ops-agent.stop

# Delete file to restore
rm /var/run/ops-agent.stop
```

**2. Signal trigger:**

```bash
# SIGUSR1 → emergency stop
kill -USR1 $(pgrep -f ops-agent)
```

**3. CLI trigger:**

```
> readonly on
```

Regardless of trigger method, the Agent will:
1. Immediately set `readonly = True`
2. Reject any L2+ operations (L0/L1 still allowed)
3. Alert in the IM channel
4. Write to audit log
5. Wait for explicit `readonly off` from a human to resume

### Blacklist

`safety.py` maintains a hard-coded list of dangerous command patterns. Commands the Agent outputs are matched against this list before execution. No matter how the LLM is "persuaded", this gate cannot be bypassed:

- `rm -rf /` / `rm -rf /*` / `rm -rf ~`
- `mkfs.*` / `dd if=`
- `DROP DATABASE` / `DROP TABLE` / `TRUNCATE`
- `shutdown` / `reboot` / `halt` / `poweroff`
- `chown -R / *` / `chmod -R 777 /`
- ...See full list in `src/safety/safety.py`

## 8. Extensions

### 8.1 Adding a Playbook

Drop a markdown file into `notebook/playbook/` and the Agent can use it on the next loop. Format is flexible; suggested structure:

```markdown
# nginx 502 Bad Gateway

## When to use me
- nginx error log shows connect() failed (111: Connection refused)
- Upstream returns 502

## Check first
- `systemctl status <upstream>` to see if backend is running
- `tail -n 100 /var/log/nginx/error.log` for specific errors
- `ss -tlnp | grep <port>` to check if port is listening

## How to fix
1. Upstream process dead → `systemctl restart <upstream>`
2. Upstream connection pool full → check `keepalive` in `/etc/nginx/nginx.conf`
3. Upstream OOM → check dmesg + adjust memory

## Verification criteria
- `curl http://localhost/` returns 200
- No new 502s in error log for 1 consecutive minute
```

The Agent finds this playbook via keyword matching during the `find_relevant` phase and injects it into the prompt.

### 8.2 Adding Git Host / Notifier Channels

`git_host.py` and `notifier.py` both use abstract base classes with multiple implementations. Adding a new channel is straightforward:

**Git Host** (see `GitHubClient` / `GitLabClient`):

```python
class GiteaClient(GitHostClient):
    def push_branch(self, repo_path, branch): ...
    def create_pr(self, repo_path, branch, base, title, body): ...
    def merge_pr(self, repo_path, pr_number): ...
    def get_pr_status(self, repo_path, pr_number): ...

# Register in factory (src/infra/git_host.py)
def make_client(host_type, run=None):
    ...
    if host_type == "gitea":
        return GiteaClient(run=run)
```

**Notifier** (see `SlackNotifier`):

```python
class TeamsNotifier(_HTTPNotifier):
    def send(self, title, content, urgency="info"):
        payload = {
            "text": f"{title}\n\n{content}",
            "themeColor": {"info": "0078D7", "warning": "FF9900",
                           "critical": "D13438"}.get(urgency),
        }
        return self._post(payload)

# Register in factory (src/infra/notifier.py)
def make_notifier(config, http_fn=None):
    ...
    if t == "teams":
        return TeamsNotifier(config.webhook_url, http_fn=http_fn)
```

## 9. Troubleshooting

### Primary Diagnostic Tool

```bash
# One-command config validation
ops-agent check

# Including LLM connectivity test
ops-agent check --test-llm
```

### Agent fails to start

```
RuntimeError: pip install anthropic
```
→ Install dependencies: `pip install -r requirements.txt`

```
ValueError: Unsupported provider: xxx
```
→ `OPS_LLM_PROVIDER` only supports `anthropic` / `openai` / `zhipu`

### Agent not responding to human input

- Check if it's in the middle of LLM streaming (terminal shows "Thinking..." prompt) — input will interrupt it immediately
- Check `pause` state: use the `status` command
- In extreme cases, `Ctrl-C` to exit and restart; state recovers from `state.json`

### Health endpoint 401/connection refused

- Port occupied? Change port: see `start_health_server` call site
- Default listens on `127.0.0.1`, not reachable remotely — this is intentional for security

### Patch apply failure

```
git apply failed (rc=1)
```

Troubleshooting order:
1. Check the `apply_output` field in the incident notes — usually the LLM-generated diff context doesn't match
2. PatchLoop auto-retries 3 times; check if all failed
3. On failure the local workspace **should** be fully rolled back. If not, manually:
   ```bash
   cd /opt/sources/backend
   git status                    # Should be clean
   git branch | grep fix/agent    # Should have no leftover branches
   git stash list                 # Check for leftover stashes
   ```

### LLM enters degraded state

```
🚨 LLM calls keep failing, switched to read-only mode
```

- Check if API key is still valid: `echo $OPS_LLM_API_KEY`
- Check network: `curl https://api.anthropic.com`
- Check balance/quota
- Agent auto-retries recovery every 5 minutes; IM channel will notify on recovery

### Notebook corruption

```bash
# Manual verification
cd notebook && git fsck

# If a remote is configured
# The Agent will try to auto-recover from remote on startup
# Manual recovery:
mv .git .git.broken
git init
git remote add origin <remote-url>
git fetch origin
git reset --hard origin/HEAD
```

### IM notifications not sending

1. Check `notifier.yaml` `type` and `webhook_url`
2. Check `notify_on` whitelist includes the trigger event
3. Check `quiet_hours` isn't in a silent period
4. Check logs: `grep notifier journalctl -u ops-agent`

## 10. Ops Best Practices

### Incremental Trust Strategy

Don't let the Agent auto-run everything from day one. Recommended progressive path:

| Phase | Duration | Configuration |
|---|---|---|
| **Observation** | 1–2 weeks | `--readonly`, just watch how the Agent diagnoses, don't let it act |
| **Assisted** | 2–4 weeks | Turn off readonly, but keep `permissions.md` strict — most actions escalate to human |
| **Autonomous L2** | Ongoing | Open up common L2 actions (restart/clean logs), disable Sprint 3-4 code auto-fix (don't configure `git_host` in `source_repos`) |
| **Autonomous L3** | Ongoing | Enable code auto-fix, but start with `max_auto_merges_per_day=2` and gradually increase |

### Notebook Backup

`notebook/` is the Agent's entire memory. Recommendations:

```bash
# 1. Configure a remote (for disaster recovery)
cd notebook
git remote add origin git@github.com:org/ops-agent-notebook.git
git push -u origin main

# 2. Enable auto-push in main.py startup args (via cron or manual trigger)
# 3. Or daily cron:
0 3 * * * cd /var/lib/ops-agent/notebook && git push origin main
```

### Lessons Distillation

After each incident is handled, the Agent writes `notebook/lessons/<incident-id>-reflection.md`. It's a good idea to review them weekly and manually distill common lessons into playbooks.

### Monitoring OpsAgent Itself

OpsAgent manages your systems, but who manages OpsAgent? Recommendations:

1. **Prometheus scraping `/metrics`**, set alerts:
   - `ops_agent_llm_degraded == 1` for 10min → critical
   - `ops_agent_uptime_seconds < 60` → process just restarted, worth checking
   - `rate(ops_agent_actions_total[5m]) > 10` → unusually high action frequency
2. **Watchdog probing `/healthz`** → restart on failure
3. **Audit log to SIEM** — `vector` / `fluent-bit` tailing `notebook/audit/*.jsonl`
4. **IM notification dual-channel** — normal notifications to Slack, critical also to PagerDuty

### Token Cost Control

The Agent records tokens for every LLM call. In `limits.yaml`:

```yaml
llm_tokens_per_hour: 200000     # When triggered, only L0 operations allowed
llm_tokens_per_day: 1000000     # When triggered, all actions force-escalated
```

Actual costs (using Claude Sonnet 4 as example):
- One diagnostic loop ≈ 5–10K input + 1–2K output ≈ $0.05–0.10
- One full code fix (including 3 retries) ≈ 30–50K tokens ≈ $0.30–0.50
- One daily report ≈ 5K tokens ≈ $0.02

### Multi-Instance Deployment (Use with Caution)

**Current v2.3.0 assumes a single process.** The following components are not distributed-safe:

- `state.json` — single file, multiple processes would overwrite each other
- `pending_events.jsonl` — append-only but pop uses rewrite with race condition
- `LimitsEngine` (`src/safety/limits.py`) — in-memory deque, independent counts per instance
- `auto_merge_timestamps` — same as above

If you must have HA deployment, either:
1. **Active-passive mode**: one primary, one standby in readonly; manual failover
2. **Redesign shared storage layer**: move the above four items to redis / etcd

Running v2.3.0 in "active-active" mode is not recommended.
